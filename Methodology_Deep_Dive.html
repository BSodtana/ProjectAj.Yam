1 Pipeline Overview — The Big Picture

Before diving into each piece, here is the full flow of your system at a glance:

DrugBank via TDC → Clean & Remove Conflicts → Cold-Drug Split (5 folds) → Build Graphs + ECFP → Siamese GAT Model → LA + DRW Loss → Evaluate

Each drug pair $(d_a, d_b)$ goes through a shared encoder that produces one embedding per drug. Those embeddings are combined into a pairwise feature vector and classified into one of 86 DDI types. The entire system is trained end-to-end with PyTorch.

Why "Cold-Drug"? In a real clinical scenario, a new drug enters the market and we need to predict its interactions with existing drugs — the model has never seen this drug before. Cold-drug evaluation simulates exactly this: test drugs are completely absent from training data.

Pair canonicalization

Drug pairs are unordered — the interaction between (Aspirin, Ibuprofen) is the same as (Ibuprofen, Aspirin). The code enforces this by sorting the two SMILES strings alphabetically:

if smiles_a <= smiles_b:
    pair = (smiles_a, smiles_b)
else:
    pair = (smiles_b, smiles_a)

This creates a unique pair_key for each drug combination, like "CC(=O)...||CC(C)...".

Conflict detection and removal

Some pairs appear with different labels in the dataset — the same drug pair is assigned to more than one interaction type. These are conflicting pairs, and keeping them would introduce label noise.

The code groups rows by their pair_key, counts how many distinct labels (label_nunique) each group has, and removes any group where label_nunique > 1:

3 Stage 2: Cold-Drug Splitting (The Key Innovation)

This is the most important methodological choice in the entire thesis. Unlike random splitting (where the same drug can appear in both train and test), cold-drug splitting ensures that every test drug is completely unseen during training.

Step-by-step: How the split works

Step A — Compute drug "degree" weights

For every unique drug in the dataset, count how many pair groups it participates in. This is its degree weight:

$$\text{weight}(d) = |\{(d, d') : (d, d') \in \text{pairs}\}| + |\{(d', d) : (d', d) \in \text{pairs}\}|$$

A drug that interacts with many other drugs has a high weight. For example, if Aspirin appears in 500 pairs, its weight is 500.

Step B — Degree-aware fold assignment

All unique drugs are sorted by weight in descending order (highest-degree drugs first). Then, using a greedy algorithm, each drug is assigned to the fold that currently has the smallest total weight:

# Pseudocode for drug-to-fold assignment
drugs = sort_by_weight_descending(all_drugs)
fold_weights = [0, 0, 0, 0, 0]  # k=5 folds

for drug in drugs:
    # Assign to the lightest fold (ties broken randomly)
    best_fold = argmin(fold_weights)
    assign(drug, best_fold)
    fold_weights[best_fold] += weight(drug)
Why degree-aware balancing? If high-degree drugs all land in one fold, that fold's test set would be disproportionately large. Greedy balancing ensures each fold has roughly equal total interaction count, producing balanced train/valid/test sizes.

Step C — Protocol S1: Building train/valid/test from folds

With $k=5$ folds numbered 0 through 4, for a given test fold $f$:

RoleWhich fold(s)?Rule for pairs
TestFold $f$At least one drug from fold $f$, the other drug from training folds
ValidationFold $(f+1) \bmod 5$At least one drug from the valid fold, the other from training folds
TrainingRemaining 3 foldsBoth drugs must be from training folds only

The mask logic in code works like this:

$$\text{train\_mask}(a,b) = (\text{fold}(a) \notin \{f, f{+}1\}) \;\wedge\; (\text{fold}(b) \notin \{f, f{+}1\})$$ $$\text{test\_mask}(a,b) = (\text{fold}(a) = f \;\wedge\; \text{fold}(b) \in \text{train\_folds}) \;\vee\; (\text{fold}(b) = f \;\wedge\; \text{fold}(a) \in \text{train\_folds})$$
Critical constraint: No leakage! Training pairs require both drugs to be from training folds. Test pairs require exactly one drug from the test fold and the other from training folds. This guarantees zero drug overlap between train and test. The code includes explicit assertions checking that pair-key sets are fully disjoint between all splits.

Step D — Guardrails and resampling

Not every random seed produces a good split. The code enforces minimum quality standards:

GuardrailThreshold
Minimum test pairs per fold5,000
Minimum test label types per fold45 (out of 86)
Train/Valid/Test all non-emptyRequired

If any fold fails these guardrails, the code tries a new randomized assignment (up to 200 attempts) using a different seed derived from the base seed: seed + 104,729 × attempt. Among all passing assignments, the one maximizing test label coverage for the selected fold is chosen.

Output format: PyTorch Geometric Data object

The final graph is stored as a torch_geometric.data.Data object:

Data(
  x       = [num_atoms, 7],       # Node feature matrix
  edge_index = [2, num_bonds×2],  # Edge connectivity (both directions)
  edge_attr  = [num_bonds×2, 5]   # Edge feature matrix
)

Edges are stored bidirectionally — if atom 0 bonds to atom 1, we store both (0→1) and (1→0). This is needed for message passing in GNNs.

Caching

Each drug's graph is computed once, then cached to disk keyed by a SHA-256 hash of its canonical SMILES. This means re-running experiments doesn't recompute graphs. The GraphCache class handles hit/miss tracking, so you know exactly how many graphs were loaded vs. freshly computed.

Why graphs instead of just SMILES strings? SMILES is a 1D text representation — it encodes the same molecule but loses the 2D/3D structural relationships. Graph representations preserve the actual connectivity pattern, allowing the GNN to learn how neighboring atoms influence each other, which is essential for predicting how two drugs interact.

5 Stage 4: ECFP Fingerprint Generation

What is ECFP?

Extended-Connectivity Fingerprints (ECFP) are a classical cheminformatics representation. Think of them as a "checklist" of chemical substructures present in a molecule.

How ECFP works — step by step

Initialize: Assign each atom an identifier based on its local properties (element type, number of neighbors, charge, etc.).
Iterate for r rounds (radius=2): In each round, each atom collects the identifiers of its neighbors and combines them with its own identifier using a hash function. This creates a new identifier that encodes a larger neighborhood. After round 1, each identifier captures the atom + its immediate neighbors. After round 2, it captures the atom + 2-hop neighborhood.
Fold into a bit vector: All the identifiers generated across all atoms and all rounds are mapped (via hashing) into a fixed-length binary vector of 2048 bits. If an identifier maps to position 734, bit 734 is set to 1.
$$\text{ECFP4}(d) \in \{0, 1\}^{2048}$$

The "4" in ECFP4 means diameter=4 (radius=2), meaning each bit encodes a substructure of up to 2 bonds radius around each atom.

ECFP code (actual implementation)

def smiles_to_ecfp(smiles, n_bits=2048, radius=2):
    mol = Chem.MolFromSmiles(smiles)
    fp = AllChem.GetMorganFingerprintAsBitVect(
        mol, radius, nBits=n_bits, useChirality=False
    )
    arr = np.zeros(n_bits, dtype=np.uint8)
    DataStructs.ConvertToNumpyArray(fp, arr)
    return arr.astype(np.float32)

ECFP projection in the model

The raw 2048-bit vector is too high-dimensional to concatenate directly with the 128-dim graph embedding. So it goes through a projection pathway:

$$\mathbf{p} = \text{Dropout}_{0.2}\Big(\text{ReLU}\big(\text{LayerNorm}(\mathbf{W}_{\text{proj}} \cdot \text{ECFP} + \mathbf{b}_{\text{proj}})\big)\Big)$$

where $\mathbf{W}_{\text{proj}} \in \mathbb{R}^{128 \times 2048}$. This compresses the 2048-bit fingerprint into a 128-dimensional dense vector.

Why use both graph + ECFP? They capture complementary information. The graph encoder learns task-specific features end-to-end (adaptive but needs data). ECFP provides stable, well-tested chemical patterns (fixed but reliable). Your ablation shows that adding ECFP to GAT improves macro PR-AUC by +6.1% — substantial.

6 Stage 5: The GNN Encoder (GAT)

What is a Graph Attention Network?

A GAT is a type of Graph Neural Network that uses attention to learn how much each neighbor should influence a node's updated representation. Instead of treating all neighbors equally, it assigns different importance weights.

GAT layer math — one layer at a time

Single attention head

For a single attention head, the computation for node $i$ works like this:

1. Linear transformation: Transform each node's features into a new space:

$$\mathbf{h}'_i = \mathbf{W} \cdot \mathbf{h}_i$$

where $\mathbf{W}$ is a learnable weight matrix and $\mathbf{h}_i$ is the current feature vector of node $i$.

2. Attention coefficients: Compute a raw attention score between node $i$ and each neighbor $j$:

$$e_{ij} = \text{LeakyReLU}\Big(\mathbf{a}^T [\mathbf{h}'_i \| \mathbf{h}'_j \| \mathbf{e}_{ij}^{\text{edge}}]\Big)$$

where $\mathbf{a}$ is a learnable attention vector, $\|$ means concatenation, and $\mathbf{e}_{ij}^{\text{edge}}$ is the 5-dim edge feature. The edge_dim=5 parameter in the code enables this — the GAT incorporates bond-type information into attention.

3. Normalize with softmax: Convert raw scores to weights that sum to 1 across all neighbors of node $i$:

$$\alpha_{ij} = \frac{\exp(e_{ij})}{\sum_{k \in \mathcal{N}(i)} \exp(e_{ik})}$$

This $\alpha_{ij}$ is the attention weight — it tells us how important neighbor $j$ is for updating node $i$.

4. Weighted aggregation: Compute the new embedding for node $i$ by taking a weighted sum of all neighbor embeddings:

$$\mathbf{h}_i^{\text{new}} = \sum_{j \in \mathcal{N}(i)} \alpha_{ij} \cdot \mathbf{h}'_j$$

Multi-head attention (4 heads)

Your model uses 4 attention heads. Each head performs the above computation independently with different learned parameters. For non-final layers, their outputs are concatenated:

$$\mathbf{h}_i^{\text{multi}} = \Big\|_{k=1}^{4} \mathbf{h}_i^{(k)}$$

If each head outputs 64-dim vectors, concatenation gives $4 \times 64 = 256$ dimensions. For the final layer, heads are averaged instead (concat=False), producing a 128-dim output.

Your 3-layer GAT architecture

LayerInput dimOutput dimHeadsConcat?Activation

Global mean pooling

After 3 GAT layers, every atom has a 128-dim embedding. But we need one embedding for the entire molecule. Global mean pooling averages all atom embeddings:

$$\mathbf{g} = \frac{1}{|\mathcal{V}|} \sum_{i \in \mathcal{V}} \mathbf{h}_i^{(\text{final})} \quad \in \mathbb{R}^{128}$$

where $\mathcal{V}$ is the set of all atoms in the molecule.

Why ELU instead of ReLU? ELU (Exponential Linear Unit) allows small negative outputs instead of hard-zeroing them like ReLU. This helps with gradient flow in GAT layers. The formula: $\text{ELU}(x) = x$ if $x > 0$, and $\alpha(e^x - 1)$ if $x \leq 0$ (with $\alpha=1$).

7 Stage 6: Siamese Pair Model & Pairwise Features

Siamese architecture

"Siamese" means the same encoder with shared weights processes both drugs. Drug A and Drug B go through the exact same GAT network:

Drug A graph → Shared GAT → g_a (128-dim)
Drug B graph → Shared GAT → g_b (128-dim)
Why shared weights? Drug A and Drug B are both just drugs — there's no inherent difference in their "role." Sharing weights means the model learns a single universal drug representation function, which is more parameter-efficient and prevents overfitting.

Drug embedding fusion (graph + ECFP)

After getting the graph embedding $\mathbf{g}_a$ (128-dim) and the projected ECFP $\mathbf{p}_a$ (128-dim), they are concatenated and fused:

$$\mathbf{h}_a = \text{Dropout}_{0.2}\Big(\text{ReLU}\big(\mathbf{W}_{\text{fuse}} [\mathbf{g}_a \| \mathbf{p}_a] + \mathbf{b}_{\text{fuse}}\big)\Big)$$

where $\mathbf{W}_{\text{fuse}} \in \mathbb{R}^{128 \times 256}$. The 256-dim concatenation is projected back down to 128 dimensions. Same for drug B → $\mathbf{h}_b$.

Pairwise feature construction (permutation-invariant)

Now we have $\mathbf{h}_a, \mathbf{h}_b \in \mathbb{R}^{128}$. The pair feature must be order-invariant — swapping drug A and B should give the same prediction. The code achieves this with:

$$\mathbf{z}_{ab} = \Big[\min(\mathbf{h}_a, \mathbf{h}_b) \;\Big\|\; \max(\mathbf{h}_a, \mathbf{h}_b) \;\Big\|\; \max - \min \;\Big\|\; \mathbf{h}_a \odot \mathbf{h}_b\Big]$$

Let's break down each component (all operations are element-wise):

ComponentDimensionWhat it captures
$\min(\mathbf{h}_a, \mathbf{h}_b)$128The "smaller" drug signal per dimension — a symmetric baseline
$\max(\mathbf{h}_a, \mathbf{h}_b)$128The "larger" drug signal per dimension — a symmetric baseline
$\max - \min$128The absolute difference — how different are the drugs?
$\mathbf{h}_a \odot \mathbf{h}_b$128Element-wise product — interaction/correlation signal

Total pair feature: $\mathbf{z}_{ab} \in \mathbb{R}^{512}$.

Why min/max instead of just concatenation? If you concatenate $[\mathbf{h}_a \| \mathbf{h}_b]$, swapping A and B gives $[\mathbf{h}_b \| \mathbf{h}_a]$, which is different! Using $\min$ and $\max$ ensures the output is identical regardless of input order. The element-wise product $\mathbf{h}_a \odot \mathbf{h}_b$ is also naturally symmetric.

Classification head

The 512-dim pair vector is classified by a 2-layer MLP:

$$\text{logits} = \mathbf{W}_2 \cdot \text{Dropout}_{0.2}\Big(\text{ReLU}\big(\mathbf{W}_1 \cdot \mathbf{z}_{ab} + \mathbf{b}_1\big)\Big) + \mathbf{b}_2$$

where $\mathbf{W}_1 \in \mathbb{R}^{256 \times 512}$ and $\mathbf{W}_2 \in \mathbb{R}^{86 \times 256}$. The output is an 86-dimensional logit vector — one value per interaction type.

8 Stage 7: Logit Adjustment — Fighting Class Imbalance

The problem

The 86 interaction types are heavily imbalanced. Some types have thousands of training examples, while rare types might have only a handful. Without correction, the model would learn to always predict common classes and ignore rare ones.

The math behind logit adjustment

First, compute the class prior — the probability of each class in the training set:

$$\pi_c = \frac{n_c + \epsilon}{\sum_{j=1}^{86}(n_j + \epsilon)}, \quad \epsilon = 10^{-12}$$

where $n_c$ is the number of training examples in class $c$. The tiny $\epsilon$ prevents division by zero for classes with zero training count.

Then compute the log prior:

$$\log \pi_c = \log\left(\frac{n_c + \epsilon}{\sum_{j}(n_j + \epsilon)}\right)$$

For a common class (high $n_c$), $\log \pi_c$ is a large negative number (close to 0). For a rare class (low $n_c$), $\log \pi_c$ is a very negative number (far from 0).

The adjusted logits are:

$$\tilde{s}_c = s_c + \tau \cdot \log \pi_c, \quad \tau = 0.5$$

Intuitive explanation

Let's work through a concrete example. Suppose:

Class$n_c$$\pi_c$$\log \pi_c$$\tau \cdot \log \pi_c$
Common (class 5)10,0000.15-1.90-0.95
Rare (class 72)500.0007-7.24-3.62

The adjustment subtracts more from common classes (by adding a bigger negative number) and subtracts less from rare classes. This effectively gives rare classes a relative boost in the softmax competition.

Why $\tau = 0.5$? $\tau$ controls the strength of the adjustment. $\tau=0$ means no adjustment. $\tau=1$ is the full theoretical correction from Menon et al. (2021). $\tau=0.5$ is a moderate middle ground — it helps rare classes without over-correcting so aggressively that it destabilizes training.

Where it's applied

The adjustment happens in two places:

(1) Inside the loss function — the adjusted logits go into cross-entropy, so the gradient signal accounts for class imbalance during backpropagation.

(2) During prediction — predict_proba() applies the same adjustment before softmax, so the output probabilities are also calibrated for class frequency.

9 Stage 8: Deferred Re-Weighting (DRW)

The key insight

Research by Cao et al. (2019) showed that applying class re-weighting from the start of training can actually hurt performance. Early in training, the model is still learning basic feature representations — if you aggressively upweight rare classes, the noisy gradients from those few samples can destabilize learning. It's better to let the model learn general features first, then shift attention to tail classes.

Two-phase training schedule

Phase 1: Epochs 1–14Phase 2: Epochs 15–20
Class weightsNone (all classes treated equally)Inverse-sqrt weights activated
Learning rate$10^{-3}$$2 \times 10^{-4}$ (5× reduction)
FocusLearn stable shared representationsFine-tune for tail classes

Class weight formula

When DRW activates at epoch 15, the loss function uses per-class weights defined as:

$$w_c^{\text{raw}} = \frac{1}{\sqrt{n_c + \epsilon}}, \quad \epsilon = 10^{-12}$$

These raw weights are then normalized so that the sample-weighted mean equals 1.0 (this prevents the overall loss magnitude from jumping when DRW activates):

$$w_c^{\text{norm}} = \frac{w_c^{\text{raw}}}{\frac{\sum_j w_j^{\text{raw}} \cdot n_j}{\sum_j n_j}}$$

Finally, weights are clipped to $[0.25, 4.0]$ to prevent extreme values:

$$w_c = \text{clip}(w_c^{\text{norm}}, 0.25, 4.0)$$

Concrete example

Class$n_c$$w_c^{\text{raw}}$After normalization & clipEffect
Common (10,000 samples)10,000$\frac{1}{\sqrt{10000}} = 0.01$~0.25 (clipped at min)Downweighted 4×
Medium (500 samples)500$\frac{1}{\sqrt{500}} = 0.045$~1.0Neutral
Rare (10 samples)10$\frac{1}{\sqrt{10}} = 0.316$~4.0 (clipped at max)Upweighted 4×

The weighted cross-entropy loss

During the DRW phase, the full loss function becomes:

$$\mathcal{L} = -\sum_{i=1}^{N} w_{y_i} \cdot \log \frac{\exp(\tilde{s}_{y_i})}{\sum_{c=1}^{86} \exp(\tilde{s}_c)}$$

where $\tilde{s}_c$ is the logit-adjusted score and $w_{y_i}$ is the class weight for the true label of sample $i$.

Learning rate reduction

At the DRW switch (epoch 15), the learning rate drops by factor drw_lr_drop = 0.2:

$$\text{lr}_{\text{new}} = \text{lr}_{\text{old}} \times 0.2 = 10^{-3} \times 0.2 = 2 \times 10^{-4}$$
Why reduce learning rate when DRW starts? The introduction of class weights changes the loss landscape. If the learning rate stays high, the model might overshoot and destabilize. A smaller learning rate allows the model to gently adjust toward tail-class improvement without forgetting what it already learned.

Sanity checks in the code

What happens in one training epoch

Check DRW phase: If the current epoch $\geq$ 15, activate class weights and drop the learning rate (only on the exact transition epoch).
For each mini-batch of 64 pairs:
(a) Move graphs, features, and labels to GPU
(b) Forward pass: compute logits from both drug graphs + fingerprints
(c) Apply logit adjustment: $\tilde{\mathbf{s}} = \mathbf{s} + 0.5 \cdot \log \boldsymbol{\pi}$
(d) Compute the objective loss (weighted CE on adjusted logits)
(e) Also compute the plain NLL loss (standard CE with no weights) for diagnostics
(f) Backpropagate and update model parameters
Evaluate on validation set: Forward pass (no gradients), compute all metrics.
Model selection: If this epoch's validation macro PR-AUC is the best so far, save the model checkpoint.
Early stopping: If macro PR-AUC hasn't improved for 5 consecutive epochs (patience=5), stop training early.

Why track two losses?

The code tracks both objective_loss (the actual loss being optimized, which includes class weights) and plain_nll_loss (standard cross-entropy without any weights). This is diagnostically important because:

When DRW activates at epoch 15, the objective loss often increases even though the model is improving — because the weights amplify the contribution of hard tail-class errors. The plain NLL loss provides an unbiased view of whether the model is actually learning better representations overall.

Model selection metric: Why macro PR-AUC?

Macro PR-AUC is ideal for imbalanced multi-class problems because it rewards the model for ranking positive examples high across all classes equally, including rare ones. Accuracy would let the model get away with always predicting common classes. Macro PR-AUC forces the model to be good at every class to score well.

11 Stage 10: Evaluation Metrics — The Full Suite

Your thesis reports 10 different metrics. Here is each one explained with its math and interpretation.

1. Accuracy

$$\text{Accuracy} = \frac{\text{Number of correct predictions}}{\text{Total predictions}} = \frac{\sum_{i=1}^{N} \mathbb{1}[\hat{y}_i = y_i]}{N}$$

Intuition: "What fraction of all predictions were correct?" Simple and interpretable, but misleading for imbalanced data — a model that always predicts the most common class could get 30%+ accuracy while being useless for rare classes.

Your result: 0.6351 mean (range 0.62–0.66 across folds)

2. Macro-F1

First, for each class $c$, compute precision and recall:

$$\text{Precision}_c = \frac{TP_c}{TP_c + FP_c}, \quad \text{Recall}_c = \frac{TP_c}{TP_c + FN_c}$$

Then compute the F1-score (harmonic mean of precision and recall) for each class:

$$F1_c = 2 \cdot \frac{\text{Precision}_c \cdot \text{Recall}_c}{\text{Precision}_c + \text{Recall}_c}$$

Finally, take the unweighted average across all classes present in the test set:

$$\text{Macro-F1} = \frac{1}{|\mathcal{C}_{\text{test}}|} \sum_{c \in \mathcal{C}_{\text{test}}} F1_c$$

Intuition: "How well does the model perform on average across all classes?" Because it's an unweighted average, a rare class with 10 samples counts just as much as a common class with 10,000 samples. This directly measures whether the model works for the full class spectrum.

Your result: 0.5032 mean — this is notably lower than accuracy because rare classes drag the average down.

3. Micro-F1

$$\text{Micro-F1} = \frac{\sum_c TP_c}{\sum_c TP_c + \frac{1}{2}(\sum_c FP_c + \sum_c FN_c)}$$

Intuition: Pools all true positives, false positives, and false negatives across classes before computing F1. This is dominated by common classes (since they contribute more TP/FP/FN). In multi-class classification, Micro-F1 equals Accuracy.

4. Cohen's Kappa ($\kappa$)

$$\kappa = \frac{p_o - p_e}{1 - p_e}$$

where $p_o$ is the observed agreement (accuracy) and $p_e$ is the expected agreement by chance:

$$p_e = \sum_{c=1}^{86} \frac{n_c^{\text{true}} \cdot n_c^{\text{pred}}}{N^2}$$

Intuition: "How much better is the model than random guessing?" If $\kappa = 0$, the model is no better than chance. If $\kappa = 1$, perfect agreement. Your $\kappa = 0.558$ means the model is substantially better than random, accounting for the class distribution.

Kappa interpretation guidelines: 0.0–0.20 = slight, 0.21–0.40 = fair, 0.41–0.60 = moderate, 0.61–0.80 = substantial, 0.81–1.00 = almost perfect. Your 0.558 falls in "moderate" to "substantial."

5. Macro ROC-AUC (One-vs-Rest)

For each class $c$, treat the problem as binary: "class $c$ vs. all other classes." Compute the ROC-AUC for this binary problem using the predicted probability $P(y=c)$:

$$\text{ROC-AUC}_c = \int_0^1 \text{TPR}_c(t) \; d\text{FPR}_c(t)$$

Then average across all classes that have both positive and negative examples in the test set:

$$\text{Macro ROC-AUC} = \frac{1}{|\mathcal{C}_{\text{scored}}|}\sum_{c \in \mathcal{C}_{\text{scored}}} \text{ROC-AUC}_c$$

Intuition: "If I pick a random positive and a random negative sample for class $c$, what's the probability the model ranks the positive higher?" A value of 0.5 is random; 1.0 is perfect. Your 0.9395 means the model has excellent discrimination ability — it can rank the correct class high even if the exact probability isn't calibrated.

Important: Classes that have zero positive or zero negative test samples are excluded from the macro average (you can't compute ROC-AUC without both). The code logs which classes are excluded, ensuring transparency.

6. Macro PR-AUC (One-vs-Rest) — Your Model Selection Metric

Similar to ROC-AUC but uses the Precision-Recall curve instead:

$$\text{PR-AUC}_c = \int_0^1 \text{Precision}_c(r) \; d\text{Recall}_c(r)$$
$$\text{Macro PR-AUC} = \frac{1}{|\mathcal{C}_{\text{scored}}|}\sum_{c \in \mathcal{C}_{\text{scored}}} \text{PR-AUC}_c$$

Intuition: PR-AUC is more demanding than ROC-AUC for imbalanced problems. ROC-AUC can be high even when the model generates many false positives for rare classes (because there are so many true negatives). PR-AUC penalizes false positives much more heavily. A random baseline has PR-AUC equal to the class prior (very small for rare classes), so your 0.5519 is meaningfully above chance.

7. Tail Macro PR-AUC

$$\text{Tail classes} = \text{bottom } 20\% \text{ of classes ranked by training support}$$

The code sorts all 86 classes by their training sample count, takes the bottom 20% (~17 classes), and computes macro PR-AUC only over those:

$$\text{Tail Macro PR-AUC} = \frac{1}{|\mathcal{C}_{\text{tail}}|}\sum_{c \in \mathcal{C}_{\text{tail}}} \text{PR-AUC}_c$$

Your result: 0.2156 mean — substantially lower than overall PR-AUC (0.5519), confirming that rare classes remain the hardest challenge.

8. Expected Calibration Error (ECE)

ECE measures whether the model's predicted confidence matches its actual accuracy. The procedure:

For each sample, take the maximum predicted probability as the confidence: $\text{conf}_i = \max_c P(y=c|\mathbf{x}_i)$
Divide samples into 15 bins based on their confidence level: [0.0–0.067], [0.067–0.133], ..., [0.933–1.0]
For each bin, compute the average confidence and the actual accuracy
ECE is the weighted average of the gap between confidence and accuracy across bins:
$$\text{ECE} = \sum_{b=1}^{15} \frac{|B_b|}{N} \left|\text{acc}(B_b) - \text{conf}(B_b)\right|$$

Intuition: If the model says "I'm 80% confident," it should be correct about 80% of the time. ECE = 0 means perfect calibration. Your ECE of ~0.20 means the model's probabilities are not well calibrated — it tends to be overconfident or underconfident. Lower is better.

Key finding: Your strongest discriminative model (GAT+ECFP+LA+DRW) has the worst ECE (0.20), while the simpler GIN-only model has the best ECE (0.12). This is the discrimination vs. calibration trade-off — a major discussion point for your defense.

9. Brier Score

$$\text{Brier} = \frac{1}{N} \sum_{i=1}^{N} \sum_{c=1}^{86} (P_{ic} - \mathbb{1}[y_i = c])^2$$

Intuition: The mean squared error between the predicted probability vector and the one-hot true label. If the model perfectly predicts $P(y=c) = 1$ for the correct class and 0 for all others, Brier = 0. It combines both discrimination and calibration into a single score. Lower is better.

Your result: ~0.566 — a composite view confirming room for improvement in probability quality.

10. Negative Log-Likelihood (NLL)

$$\text{NLL} = -\frac{1}{N} \sum_{i=1}^{N} \log P(y = y_i | \mathbf{x}_i)$$

Intuition: "How surprised is the model by the true labels?" If the model assigns high probability to the correct class, $-\log P$ is small. If the model assigns low probability (it's surprised), $-\log P$ is large. This is a proper scoring rule — the only way to minimize it is to predict the true probabilities. Lower is better.

12 Putting It All Together — End-to-End Flow

Let's trace a single prediction through the entire system:

Input: A drug pair

drug_a = "CC(=O)Oc1ccccc1C(=O)O"   # Aspirin
drug_b = "CC12CCC3C(C1CCC2O)CCC4=..."  # Some steroid
true_label = 42

Step-by-step forward pass

SMILES → Molecular Graph: RDKit parses each SMILES into a graph. Aspirin has ~13 atoms (nodes) with 7 features each, and ~13 bonds (edges × 2 directions) with 5 features each.
SMILES → ECFP4: RDKit computes a 2048-bit fingerprint for each drug. Aspirin's fingerprint encodes substructures like "aromatic ring," "ester group," "carboxylic acid," etc.
GAT encoding (shared weights): Both graphs pass through the same 3-layer GAT. After 3 rounds of attention-weighted message passing and global mean pooling, we get $\mathbf{g}_a, \mathbf{g}_b \in \mathbb{R}^{128}$.
ECFP projection: Each 2048-bit fingerprint is projected to 128 dims: $\mathbf{p}_a, \mathbf{p}_b \in \mathbb{R}^{128}$.
Fusion: Graph and ECFP embeddings are concatenated and fused: $\mathbf{h}_a = \text{MLP}([\mathbf{g}_a \| \mathbf{p}_a]) \in \mathbb{R}^{128}$. Same for drug B.
Pair features: The symmetric pair vector is constructed: $\mathbf{z}_{ab} = [\min \| \max \| \text{diff} \| \text{product}] \in \mathbb{R}^{512}$.
Classification: The MLP classifier produces 86 raw logits: $\mathbf{s} \in \mathbb{R}^{86}$.
Logit adjustment: $\tilde{\mathbf{s}} = \mathbf{s} + 0.5 \cdot \log \boldsymbol{\pi}$.
Softmax → probabilities: $P(y=c) = \frac{\exp(\tilde{s}_c)}{\sum_j \exp(\tilde{s}_j)}$ for each class $c$.
Prediction: $\hat{y} = \arg\max_c P(y=c)$. If $\hat{y} = 42$, the prediction matches the true label!

During training (backward pass)

The loss is computed as:

$$\mathcal{L} = -w_{42} \cdot \log P(y=42)$$

where $w_{42}$ is the class weight for type 42 (if DRW is active). Gradients flow backward through the classifier, fusion MLP, ECFP projection, all 3 GAT layers, updating every learnable parameter.

Total parameter flow: ECFP is fixed (not learned) — it always provides the same fingerprint for the same molecule. Everything else is learned end-to-end: GAT attention weights, GAT linear transforms, ECFP projection, fusion MLP, and the classifier. The logit adjustment uses fixed log-priors computed once from the training set.
